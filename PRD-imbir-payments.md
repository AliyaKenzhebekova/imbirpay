## 1. Обзор продукта

### 1.1 Описание

**ImbirPay** — внутренняя веб-система управления заявками на оплату для ТОО «Imbir Group». Обеспечивает прозрачный маршрут от подачи заявки инициатором до исполнения казначеем через двухуровневое согласование: Учредитель → Финансовый директор → Казначей.

### 1.2 Проблема

До внедрения системы заявки поступали в казначейство неструктурированно: через мессенджеры, email, устно. Это создавало:
- Дублирование платежей по одному счёту
- Отсутствие прозрачности статуса заявки
- Невозможность контролировать суммы по статусам
- Нет разделения безналичных платежей и выдачи подотчёт
- Нет зафиксированной цепочки согласования

### 1.3 Решение

Закрытая веб-система с авторизацией, двумя типами заявок, четырёхшаговой формой, автоматической маршрутизацией и казначейской панелью с KPI по суммам.

## 3. Функциональные требования

### 3.1 Авторизация

Система закрыта — доступ только для сотрудников ТОО «Imbir Group».

| Способ входа | Описание |
|---|---|
| **Google OAuth** | Корпоративный аккаунт @imbir.kz |
| **Face ID** | Биометрическая аутентификация |
| **Email + пароль** | Резервный способ |

После аутентификации пользователь выбирает роль. Роль определяет отображаемый интерфейс и доступные действия.

---

### 3.2 Два типа заявок

#### Тип 1: Оплата счёта (безналичный платёж)

Используется для платежей поставщикам по выставленному счёту.

**Идентификатор:** имя загруженного файла счёта (PDF/JPG/PNG)

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Тип заявки | ✓ | «Оплата счёта» |
| Категория расходов | ✓ | Аренда / Поставщики / Услуги / Коммунальные / Зарплата / Прочее |
| Скан счёта (файл) | ✓ | PDF, JPG, PNG — до 10 МБ. Реквизиты извлекает казначей |
| Сумма (тенге) | ✓ | Берётся из загруженного счёта |
| Желаемая дата оплаты | ✓ | |
| Приоритет | ✓ | Плановая / Стандарт / Срочно (ползунок) |
| Назначение и обоснование | ✓ | Объединённое поле: цель + почему важно оплатить вовремя |

**Дедупликация:** система блокирует отправку, если файл с таким именем уже зарегистрирован

#### Тип 2: Выдача подотчёт (наличные / аванс)

Используется для получения средств на хозяйственные, командировочные и прочие нужды.

**Идентификатор:** уникальное название заявки + сумма

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Тип заявки | ✓ | «Выдача подотчёт» |
| Категория расходов | ✓ | Из общего списка категорий |
| Название / номер заявки | ✓ | Уникальный идентификатор (напр. ЗП-2026-001) |
| Получатель (ФИО) | ✓ | Кому выдаются средства |
| Форма выдачи | — | Наличные / Перевод на карту / Аванс |
| Период использования | ✓ | Даты + срок отчёта |
| Сумма (тенге) | ✓ | Запрашиваемая сумма |
| Желаемая дата выдачи | ✓ | |
| Приоритет | ✓ | Ползунок |
| Цель расходов и обоснование | ✓ | Единое поле |

**Дедупликация:** система блокирует, если заявка с таким именем уже существует

### 3.3 Четырёхшаговая форма подачи заявки

```
Шаг 01         Шаг 02              Шаг 03           Шаг 04
Инициатор  →   Счёт / Подотчёт  →  Маршрут      →  Подтверждение
```

#### Шаг 01 — Инициатор

- Выбор филиала (из списка 8 филиалов)
- Выбор отдела
- ФИО инициатора (выбор из списка → должность подставляется автоматически)

#### Шаг 02 — Данные заявки

- **02·A** — Тип заявки (карточки: Оплата счёта / Выдача подотчёт)
- **02·B** — Категория расходов (6 карточек)
- **02·C** — Специфичные поля по типу (скан счёта ИЛИ данные подотчёта)
- **02·D** — Сумма, дата, приоритет, назначение+обоснование

#### Шаг 03 — Маршрут согласования

Показывает: краткую сводку заявки + цепочку согласования с именами (Мусина Дина → Алия Кенжебекова → Гульнар Сулейменова)

#### Шаг 04 — Итоговое подтверждение

Таблица со всеми данными заявки + чекбокс согласия + кнопка отправки

**После отправки:** заявка автоматически направляется руководителю (статус → «У руководителя»). Присваивается уникальный номер IMG-ГГГГ-XXXXX.

---

### 3.4 Цепочка согласования

```
Инициатор подаёт
       ↓
👔 Мусина Дина (Учредитель) — 1-е согласование
       ↓ одобрено
💼 Алия Кенжебекова (Финансовый директор) — 2-е согласование
       ↓ одобрено
🏦 Гульнар Сулейменова (Казначей) — Исполнение оплаты
       ↓
💳 Статус: Оплачено
```

На каждом этапе: одобрить ✅ или отклонить ✕. При отклонении цепочка прерывается, статус → «Отклонено».

**Ускоренный режим:** Руководитель, Финдиректор и Казначей имеют полный доступ — могут продвигать любую заявку по цепочке вне зависимости от её текущего статуса.

### 3.5 Казначейская панель

#### KPI-блок (6 карточек)

Каждая карточка содержит: статус, количество заявок, **общую сумму в ₸**, подпись

| Карточка | Цвет | Сумма |
|---|---|---|
| Новые заявки | Жёлтый | Σ сумм новых |
| У руководителя | Жёлтый | Σ на первом согласовании |
| У финдиректора | Жёлтый | Σ на втором согласовании |
| Одобрено | Зелёный | Σ готовых к оплате |
| Оплачено | Синий | Σ исполненных |
| Отклонено | Красный | Σ отклонённых |

#### Фильтры

Все / Новые / У руководителя / У финдиректора / Одобрено / Оплачено / Отклонено + поиск по тексту + кнопка «Выгрузить CSV»

#### Таблица заявок

Колонки: тип (🧾/💵), ID + дата + DUP-метка, ФИО + должность инициатора, назначение, вложенный файл / название подотчёта, **сумма**, категория, срок, статус, действия (✅ / ✕ / →)

#### Боковая панель детали (Drawer)

Открывается по клику на строку:
- Тип + сумма крупно
- Цепочка согласования с именами, датами и статусами
- Инициатор, должность, филиал, категория, дата, приоритет
- Для счёта: вложенный файл
- Для подотчёта: получатель, форма выдачи, период
- Назначение/обоснование
- История изменений (timeline)
- Поле комментария казначея
- Кнопки: «Одобрить» / «Отклонить» / «Скачать»

#### Массовые операции

Выбор нескольких заявок через чекбоксы → «Одобрить все» / «Отклонить» / «Выгрузить»

#### Экспорт

CSV с полями: ID, Инициатор, Филиал, Сумма, Срок, Статус, Назначение

### 3.6 Дедупликация

| Тип заявки | Идентификатор дубликата |
|---|---|
| Оплата счёта | Имя загруженного файла |
| Выдача подотчёт | Название/номер заявки |

Поведение системы:
- Предупреждение в реальном времени при вводе/загрузке
- Блокировка отправки при наличии дубликата
- В таблице дублирующие заявки помечаются меткой **DUP**

## 4. Нефункциональные требования

### 4.1 Безопасность

- Доступ только через аутентификацию (Google / Face ID / Email+пароль)
- Система закрытая — публичного доступа нет
- Данные шифруются по стандарту AES-256

### 4.2 Производительность

- Первый экран загружается менее чем за 2 секунды
- Переключение между разделами — мгновенное (< 100 мс)
- Поддержка офлайн-режима (PWA + Service Worker)

### 4.3 Устройства

- Приоритет: десктоп (рабочие компьютеры)
- Поддержка: планшет, мобильный
- Минимальный viewport: 360px

### 4.4 Хранение данных

| Версия | Хранилище |
|---|---|
| MVP v1 | localStorage на устройстве |
| v2 | Облачная БД (Supabase / Firebase) |
| v3 | Интеграция с 1С / ERP |

## 7. Статусы заявок

```
new  →  manager  →  cfo  →  approved  →  paid
 ↓        ↓          ↓
                  rejected
```

| Статус | Метка | Описание |
|---|---|---|
| `new` | 🔵 Новая | Только создана, ещё не взята в работу |
| `manager` | 👔 У руководителя | На первом согласовании у Мусиной Дины |
| `cfo` | 💼 У финдиректора | На втором согласовании у Кенжебековой |
| `approved` | ✅ Одобрено | Одобрено, ожидает исполнения казначеем |
| `paid` | 💳 Оплачено | Казначей исполнил платёж |
| `rejected` | ❌ Отклонено | Отклонено на любом этапе |
